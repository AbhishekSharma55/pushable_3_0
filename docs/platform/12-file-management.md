# File Management

The file management system provides S3-compatible storage for workspace files. Files can be uploaded by users, generated by agents, or attached during chat. The system includes folder organization, storage quotas, and a file preview UI.

---

## Architecture

```
Upload Source
  ├── Chat attachments     (source: "chat_upload")
  ├── Agent-generated files (source: "agent_generated")
  └── API/UI uploads        (source: "api_upload")
        │
        ▼
  Bucket Service
  ├── Validate (size, extension, quota)
  ├── Generate storage key
  ├── Write to MinIO/S3
  └── Create database record
        │
        ▼
  MinIO (dev) / AWS S3 (prod)
  ├── Bucket: pushable-bucket
  └── Key: workspaceId/uuid-filename
```

---

## Storage Backend

### MinIO (Development)

- **Image:** `minio/minio:latest`
- **API port:** 9000
- **Console port:** 9001
- **Default credentials:** `minioadmin` / `minioadmin`
- **Bucket:** `pushable-bucket` (auto-created on first use)

### S3 Configuration

| Variable | Default | Description |
|----------|---------|-------------|
| `S3_ENDPOINT` | `http://localhost:9000` | S3 API endpoint |
| `S3_ACCESS_KEY` | `minioadmin` | Access key |
| `S3_SECRET_KEY` | `minioadmin` | Secret key |
| `S3_BUCKET` | `pushable-bucket` | Bucket name |
| `S3_REGION` | `us-east-1` | AWS region |

The storage client uses `@aws-sdk/client-s3` with `forcePathStyle: true` for MinIO compatibility.

---

## File Limits

| Limit | Value | Configurable |
|-------|-------|-------------|
| **Max file size** | 50 MB | Hardcoded |
| **Max storage per workspace** | 500 MB | Via `MAX_BUCKET_STORAGE_MB` env var |
| **Blocked extensions** | `.exe`, `.bat`, `.cmd`, `.sh`, `.ps1`, `.msi`, `.dll`, `.so`, `.dylib` | Hardcoded |

## Database Schema

```sql
bucket_files
  ├── id          UUID (primary key)
  ├── workspaceId UUID (FK → workspaces, cascade delete)
  ├── filename    TEXT (original filename)
  ├── storageKey  TEXT (unique, S3 object key)
  ├── mimeType    TEXT
  ├── sizeBytes   BIGINT
  ├── folder      TEXT (e.g., "/", "/reports", "/chat-uploads")
  ├── source      ENUM: chat_upload, agent_generated, api_upload
  ├── sessionId   UUID (nullable, FK → sessions)
  ├── agentId     UUID (nullable, FK → agents)
  ├── uploadedBy  UUID (nullable, user who uploaded)
  ├── metadata    JSONB (optional extra data)
  ├── createdAt   TIMESTAMP
  └── updatedAt   TIMESTAMP
```

## API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/api/bucket/files` | List files (with filtering) |
| `GET` | `/api/bucket/files/:id` | Get file metadata |
| `POST` | `/api/bucket/files/upload` | Upload files (multipart) |
| `GET` | `/api/bucket/files/:id/download` | Download file binary |
| `DELETE` | `/api/bucket/files/:id` | Delete a file |
| `PUT` | `/api/bucket/files/:id` | Rename or move a file |
| `GET` | `/api/bucket/folders` | List all folders |
| `POST` | `/api/bucket/folders` | Create a folder (no-op, folders are implicit) |
| `GET` | `/api/bucket/usage` | Get storage usage stats |

### List Files

```
GET /api/bucket/files?folder=/reports&source=agent_generated&search=report&limit=50&offset=0
```

Query parameters:
- `folder` -- Filter by folder path
- `source` -- Filter by source type
- `search` -- Search in filename
- `limit` -- Max results (default: 100)
- `offset` -- Pagination offset

### Upload Files

```
POST /api/bucket/files/upload
Content-Type: multipart/form-data

folder: /reports
file: <binary>
file: <binary>  (multiple files supported)
```

### Download File

```
GET /api/bucket/files/:id/download
```

Returns the file binary with appropriate `Content-Type` and `Content-Disposition` headers.

### Rename / Move File

```json
PUT /api/bucket/files/:id
{
  "filename": "new-name.pdf",
  "folder": "/new-folder"
}
```

### Storage Usage

```
GET /api/bucket/usage
```

Returns:
```json
{
  "data": {
    "totalBytes": 52428800,
    "fileCount": 42
  }
}
```

## Upload Flow

### 1. Validation

```
1. Check file size ≤ 50 MB
2. Check extension not in blocked list
3. Check workspace storage quota:
   current usage + file size ≤ MAX_BUCKET_STORAGE_MB (500 MB)
```

### 2. Storage

```
1. Generate storage key: workspaceId/uuid-filename
2. Sanitize folder path (remove .., normalize slashes)
3. Write buffer to S3/MinIO via PutObject
4. Create database record with metadata
5. If DB insert fails → delete the S3 object (rollback)
```

### 3. Sources

| Source | When | Folder |
|--------|------|--------|
| `chat_upload` | User attaches file in chat | `/chat-uploads` |
| `agent_generated` | Agent uses `bucket_save_file` tool | `/agent-output` (default) |
| `api_upload` | User uploads via Files page or API | User-specified or `/` |

## Agent File Tools

Agents interact with files via bucket tools (see [Agent Tools](./06-agent-tools.md)):

| Tool | Description |
|------|-------------|
| `bucket_save_file` | Save a file (text or base64 binary) |
| `bucket_read_file` | Read file contents |
| `bucket_list_files` | List files in the workspace |
| `bucket_delete_file` | Delete a file |

### bucket_save_file Example

```
Agent saves a CSV report:
  filename: "monthly-report.csv"
  content: "name,email\nJohn,john@example.com"
  encoding: "text"
  folder: "/reports/2026"
  description: "Monthly lead report"
```

The tool auto-detects MIME type from the file extension.

## Folder System

Folders are **implicit** -- they exist when files use them. There is no separate folder table.

- `GET /api/bucket/folders` scans all files and returns unique folder paths
- `POST /api/bucket/folders` is a no-op (returns the path, for frontend UX)
- The root folder `/` always exists
- Folder paths are sanitized: no `..`, normalized slashes, must start with `/`

## File Preview (Frontend)

The artifact panel in the chat interface and the Files page support previewing multiple file types:

| File Type | Preview Component | Library |
|-----------|------------------|---------|
| HTML | `html-preview.tsx` | iframe sandbox |
| Markdown | `markdown-preview.tsx` | react-markdown + remark-gfm |
| CSV | `csv-preview.tsx` | Custom table renderer |
| XLSX | `xlsx-preview.tsx` | xlsx library |
| PDF | `pdf-preview.tsx` | Custom PDF viewer |
| Text | `txt-preview.tsx` | Pre-formatted text |

All preview components support:
- **Download** -- Download the original file
- **Copy to clipboard** -- Copy file content

### Artifact Panel

When an agent generates or references a file during chat, it appears in the artifact panel alongside the conversation. Users can view, download, and interact with the file without leaving the chat.

## Chat File Attachments

When files are attached to chat messages:

1. Files are processed by `fileProcessingService`:
   - Images → base64 data URLs (for multimodal LLM input)
   - PDFs → extracted text (via pdfjs-dist)
   - DOCX → extracted text (via mammoth)
   - CSV/MD/TXT → raw text
2. Files are persisted to the workspace bucket (`/chat-uploads` folder)
3. Attachment metadata is stored in the message's `metadata.attachments` field
4. The agent receives:
   - Document text prepended to the user message
   - Image data URLs as multimodal content blocks (if model supports vision)

---

## Storage Cleanup

- When a file is deleted via API, both the S3 object and database record are removed
- When a workspace is deleted, all `bucket_files` records cascade-delete, but S3 objects may need manual cleanup
- File deletion from S3 is best-effort (logged on failure, not blocking)
